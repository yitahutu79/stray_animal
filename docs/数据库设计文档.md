# 数据库设计文档

## 1. 数据库概述

- **数据库名称**：`stray_animal_db`
- **字符集**：`utf8mb4`
- **排序规则**：`utf8mb4_general_ci`
- **数据库引擎**：InnoDB

## 2. 数据库概念结构设计（E-R图说明）

### 2.1 实体说明

#### 用户实体（User）
- **属性**：用户ID、用户名、密码、真实姓名、手机号、邮箱、地址、用户类型、状态、创建时间、更新时间
- **用户类型**：ADMIN（管理员）、STAFF（工作人员）、VOLUNTEER（志愿者）、ADOPTER（领养者）

#### 动物实体（Animal）
- **属性**：动物ID、名称、种类、品种、性别、年龄、体重、颜色、健康状况、是否免疫、是否绝育、性格特征、状态、描述、创建时间、更新时间
- **状态**：待领养、领养中、已领养、特别照顾

#### 领养申请实体（AdoptionApplication）
- **属性**：申请ID、申请人、动物、申请状态、申请理由、居住环境、是否有其他宠物、收入水平、养宠经验、匹配评分、驳回原因、创建时间、更新时间
- **状态**：SUBMITTED（已提交）、UNDER_REVIEW（审核中）、APPROVED（已通过）、REJECTED（已驳回）、CANCELLED（已取消）

#### 捐赠实体（Donation）
- **属性**：捐赠ID、捐赠人、捐赠人姓名、捐赠类型、金额、状态、是否公开、备注、创建时间、更新时间
- **类型**：MONEY（资金）、ITEM（物资）
- **状态**：PENDING（待确认）、CONFIRMED（已确认）、USED（已使用）、PARTLY_USED（部分使用）

#### 捐赠使用去向实体（DonationUsage）
- **属性**：使用记录ID、所属捐赠、对应动物、项目名称、使用时间、使用说明、凭证照片、创建时间、更新时间

#### 回访记录实体（AdoptionFollowup）
- **属性**：回访ID、对应申请、执行工作人员、回访时间、回访方式、回访结果、备注、创建时间、更新时间
- **回访方式**：PHONE（电话）、HOME（上门）、ONLINE（线上）

### 2.2 实体关系说明

1. **用户 → 领养申请**：一对多（一个用户可以提交多个领养申请）
2. **动物 → 领养申请**：一对多（一个动物可以收到多个申请）
3. **用户 → 捐赠**：一对多（一个用户可以多次捐赠）
4. **捐赠 → 捐赠使用去向**：一对多（一笔捐赠可以有多次使用记录）
5. **领养申请 → 回访记录**：一对多（一个申请可以有多次回访）
6. **用户 → 回访记录**：一对多（一个工作人员可以执行多次回访）
7. **动物 → 捐赠使用去向**：多对一（一个动物可以关联多条使用记录）

## 4. 数据库物理结构设计

### 4.1 存储引擎选择
所有表均使用 **InnoDB** 存储引擎，支持事务、外键约束和行级锁。

### 4.2 字符集选择
- 数据库、表、字段均使用 `utf8mb4` 字符集，支持完整的 Unicode 字符（包括 emoji）
- 排序规则使用 `utf8mb4_general_ci`

### 4.3 索引设计原则
1. **主键索引**：所有表的主键使用自增 BIGINT
2. **唯一索引**：用户名等唯一字段建立唯一索引
3. **普通索引**：外键字段、常用查询字段（如状态、类型）建立索引
4. **复合索引**：根据实际查询场景建立（如状态+评分）

### 4.4 数据完整性约束
- **主键约束**：每个表都有主键
- **外键约束**：建立表间关联关系，保证数据一致性
- **非空约束**：关键字段设置 NOT NULL
- **默认值**：部分字段设置默认值（如状态字段）

## 7. 数据库优化建议

1. **定期备份**：建议每天备份数据库
2. **索引优化**：根据实际查询情况调整索引
3. **分区表**：如果数据量很大，可考虑按时间分区
4. **读写分离**：高并发场景可考虑主从复制
5. **缓存策略**：对热点数据（如动物列表）使用 Redis 缓存
